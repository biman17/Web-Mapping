FEEDBACK RECIEVED

Working group 1 was assigned for the peer review process of this website. The review was on the draft version of the website, inline with the rubric provided by the faculty.



No Criteria Grade Feedback Response
1 Description of the photogrammetric workflow from input to output 10 Clear and precise description of the photogrammetric process. The source of data is not specified. As per the requirements of the assignment,the web page contains more information than required. More information was provided as a different dataset was used to generate the results, so it seemed better to add more detail on it.
2 Description of the photogrammetric product in terms of quality and comparison with existing geo-information 9 The comparison with existing geo-information seems unclear. To be precise , How are the check points generated ? The check points were mannualy collected from DTMaster one of the module from Inpho software (https://geospatial.trimble.com/products-and-solutions/inpho)
3 Usage of single or multiple layer spatial data in a web mapping service 9 The requirement is fullfilled. The group provided multiple layer spatial data: two layers generated (DSM and DTM). The layers can also be selected/deselected.However,the need of contour maps for DSM seems unlikely. The countour lines for the DSM model were extracted to give a better understanding of the height variations between it and the DTM model.
4 Evaluation, analysis and visualization of the quality of the product 9 Interactive maps and good visualisation.Static maps are good and self explanatory. Rendering of the layers take time.When zoomed in,the DTM and DSM layers seem blurry. Feature information visible when we hover the mouse over the layer which enhances the visualisation process. Few high resolution images put a constraint on the server. We fixed the issue.
5 Description of web-service requirements and open standards used 10 Description and explanation on web architecture is present. The description was done to make explicit the process used to create the website.




FEEDBACK PROVIDED

We were assigned to review the website of working group 3. Below are the comments given from our group based on rubric.


SSYZ's website


No Criteria Grade Strengths Ideas for Improvement
1 The final website includes evidence that existing tools have been correctly applied by the project team to generate a meaningful photogrammetric product. 10
  • Clear and organised flowchart.
  • Expected raster layers of products visualized on the website.
  • Hill shade layers are a good addition.
  • Conditional statement in the flowchart : “Content of the block file is checked”, it is expected for orientation check and not DTM, DSM or orthophoto.
  • Two separate workflows are present in the website in different places. Maybe putting them in one place gives a clear explanation of the process.
  • Unclear study area description. Two different areas are represented in the website, but there are no clear directions to follow their visualization.
2 The final website includes evidence that the project team has autonomously and correctly produced a photogrammetric product and compared this with existing geo-information. 9 The rasters generated overlays correctly with base layers used. Instead of displaying coordinates in a metric system, a geographic coordinate system gives a better context about the study area and helps in comparison.
3 The final website includes evidence that the project team has correctly produced a web mapping service using at least one spatial data layer. 10
  • Mapserver API is used to render the photogrammetric products in the website.
  • Options given to choose base layers using OpenLayers.
  • Nice addition controls like visibility and opacity.
  • Good work in using Cesium for 3D terrain visualization - Excellent.
  • Static image as a legend doesn’t match with the existing standards of your website. Using GetLegendGraphic method to render the legend from mapserver would be a great addition and brings uniformity.
  • Mapserver and the legend color ramps don't match with each other. Previous comment covers the suggestion for this as well.
  • Generic workflow for web mapping looks good but specific to this website is more informative.
  • Two study area rasters i.e., Netherlands and Germany, in one single container without any description and documentation creates confusion among the viewers. Maybe separate containers or any other way to point out the whole design would be more desirable.
4 The final website includes evidence that the project team correctly and clearly analysed and visualized the uncertainty in the product. In the final description, the project team clearly describes the sources of uncertainty following from all choices (assumptions) the team has explicitly made. NA No uncertainty or quality assessment found in the website.
5 The project team has worked autonomously in all aspects of the group work. 10 Indication of roles is informative and clearly indicates the technical abilities of the team members.
6 The project team has provided effective feedback to other project groups and have shown that they are able to correctly deal with given feedback. NA Out of scope, hence no comments.
7 There is a complete description of web-service requirements and open standards, and the project website reflects the stated requirements. 7.5 The website uses Web Mapping Service as OWS inline with OGC.
  • Though the code could be viewed using the developer tools, an exclusive repository like GitHub would be a good way of presenting this website as open source.
  • Mapserver and the legend color ramps don't match with each other. Previous comment covers the suggestion for this as well.
  • As a working group of the same assignment, we can assume that the softwares used could be open source. But explicit mention about it on the website would confer it officially.
